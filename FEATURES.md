## Ключевые возможности и технологии проекта

- **Назначение проекта**
  - Бэкенд‑сервис и Telegram‑бот для трекинга привычек и задач.
  - Пользователь формулирует привычку по формуле: «я буду _действие_ в _время_ в _место_», а система напоминает о ней в Telegram.

- **Технологический стек**
  - **Django 5**: современный веб‑фреймворк, на котором построено основное приложение.
  - **Django REST Framework (DRF)**: реализация REST API для работы с пользователями и привычками.
  - **JWT‑аутентификация (djangorestframework-simplejwt)**: безопасная авторизация через access/refresh токены.
  - **PostgreSQL** как основная СУБД.
  - **Celery + Redis**: фоновые и периодические задачи (отправка напоминаний).
  - **Telegram Bot API (requests)**: интеграция с Telegram для доставки уведомлений.
  - **drf-spectacular**: автогенерация OpenAPI‑схемы и документации.
  - **django-environ**: конфигурация через переменные окружения и `.env` файл.

- **Модель пользователя**
  - **Кастомная модель `User`** с авторизацией по email (без `username`).
  - Поле `telegram_chat_id` хранит идентификатор Telegram‑чата для адресной отправки уведомлений.
  - Готовность к расширению (добавление полей профиля, ролей и т.п.).

- **Домены и бизнес‑логика привычек**
  - Модель `Habit` реализует привычку по формуле: «я буду _action_ в _time_ в _place_».
  - Валидация бизнес‑ограничений на уровне модели:
    - Нельзя одновременно указать и текстовое вознаграждение, и связанную «приятную» привычку.
    - Время выполнения ≤ 120 секунд.
    - Периодичность от 1 до 7 дней.
    - Связанной может быть только приятная привычка.
    - У приятной привычки не может быть отдельного вознаграждения.
  - Поддержка **публичных привычек** (`is_public`), которые могут быть видны другим пользователям.

- **Интеграция с Telegram**
  - Отдельное приложение `telegram_bot` с задачей Celery `send_due_habits_reminders`.
  - Ежеминутная/регулярная проверка привычек, у которых «наступило время».
  - Группировка привычек по пользователям и отправка **одним сообщением** в Telegram.
  - Работа с Telegram Bot API через HTTP‑запросы (`requests`) с обработкой ошибок сети.

- **Архитектура и API**
  - Разделение на приложения: `users`, `habits`, `telegram_bot`, `config`.
  - Использование **DRF‑serializers**, **permissions** и **viewsets** для чистой и расширяемой архитектуры API.
  - Стандартизированная пагинация (`PageNumberPagination`) и базовые права доступа (`IsAuthenticated` по умолчанию).
  - Подготовка к подключению SPA‑фронтенда (CORS‑настройки и описание API).

- **Надёжность и качество кода**
  - Типизация Python (аннотации типов во всех ключевых модулях).
  - Тесты (pytest + pytest‑django) для пользователей и привычек, проверка бизнес‑логики.
  - Настроено покрытие кода (coverage) с порогом `fail_under = 80`.
  - Линтеры и форматтеры: **flake8** и **black** с едиными настройками стиля.

- **Конфигурация и развертывание**
  - Все чувствительные настройки (секреты, токены, параметры БД) выносятся в `.env`.
  - Поддержка разных окружений (dev/prod) за счёт переменных `DEBUG`, `ALLOWED_HOSTS`, `CORS_ALLOWED_ORIGINS`.
  - Возможность горизонтального масштабирования за счёт использования Celery + Redis и PostgreSQL.

- **Чем проект технологичен**
  - Проработанная доменная модель привычек с жёсткой бизнес‑валидацией.
  - Интеграция с внешним сервисом (Telegram) через асинхронную очередь задач.
  - Зрелый стек для продакшен‑бэкенда (Django/DRF, JWT, Celery, Redis, PostgreSQL).
  - Стандартизированная документация API и подготовка к SPA‑клиенту.
  - Фокус на качестве кода и контроле изменений через тесты, покрытие и линтеры.
